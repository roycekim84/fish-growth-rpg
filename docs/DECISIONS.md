# 결정 기록

중요한 기술·기획 결정을 짧게 누적한다. 기존 결정을 바꿀 때는 이전 기록을 삭제하지 않고 새 항목으로 대체 이유를 남긴다.

## D-001 — Flutter + Flame 사용

- 상태: 확정
- 결정: Flutter stable 위에 Flame을 사용한다.
- 이유: Android, iOS, Web을 단일 Dart 코드베이스로 지원하면서 실시간 게임 루프, 컴포넌트, 카메라, 충돌과 스프라이트 기능을 활용할 수 있다.
- 영향: UI는 Flutter overlay, 게임 월드는 Flame 중심으로 분리한다.

## D-002 — 모바일 우선, 웹은 개발·QA 보조

- 상태: 확정
- 결정: 최종 품질 기준은 Android/iOS이며 Web은 빠른 반복 테스트와 공유 가능한 데모에 사용한다.
- 이유: 웹과 모바일은 방향 고정, 오디오 캐시, 입력 및 성능 특성이 일부 다르다.
- 영향: 웹 통과만으로 모바일 마일스톤을 완료하지 않는다.

## D-003 — 세로 화면 전용

- 상태: 확정
- 결정: 플레이 화면은 portrait only로 설계한다.
- 이유: 한 손 중심의 단순한 모바일 조작과 HUD 배치를 유지한다.
- 영향: 넓은 웹 화면에서도 세로형 캔버스를 유지한다.

## D-004 — 전체 픽셀아트 스타일

- 상태: 확정
- 결정: 배경, 물고기, 이펙트, 버튼과 패널을 픽셀아트로 통일한다.
- 이유: 작은 모바일 화면에서 강한 실루엣과 일관된 제작 범위를 확보한다.
- 영향: 에셋 규격, 필터링, atlas 여백과 픽셀 정렬을 초기부터 검증한다.

## D-005 — 문서와 Git 기반 단계 진행

- 상태: 확정
- 결정: 명세, 로드맵, 결정, 작업 현황을 Markdown으로 관리하고 마일스톤별 작은 커밋으로 진행한다.
- 이유: 범위 확대와 구현 방향 흔들림을 줄이고 변경 이유를 추적한다.
- 영향: 코드 변경과 함께 관련 문서를 갱신한다.

## D-006 — 패키지 버전 고정

- 상태: 확정
- 결정: Flutter 앱의 `pubspec.lock`을 Git에 포함한다.
- 이유: 앱 프로젝트에서 개발/CI/출시 빌드의 의존성 재현성을 확보한다.
- 영향: 의존성 업그레이드는 별도 커밋과 회귀 테스트를 거친다.

## D-007 — 앱 이름과 식별자

- 상태: 확정
- 결정: 프로젝트명은 `fish_growth_rpg`, 임시 표시명은 `Fish Growth RPG`, Android application ID와 iOS bundle ID는 `com.roycekim.fishgrowthrpg`를 사용한다.
- 이유: M1 프로젝트 생성과 양 플랫폼 설정에 사용할 안정적인 기본값이 필요하다.
- 영향: 실제 출시 이름을 바꾸더라도 패키지 식별자는 기존 설치 및 스토어 등록과 연결되므로 신중하게 변경한다.

## D-008 — 드래그 이동과 1차 부스터 정책

- 상태: 확정
- 결정: 최초 터치 지점을 앵커로 삼아 드래그 방향으로 가속하며, 손을 떼면 추진을 중단하고 관성 감속한다. 부스터는 버튼을 누르는 동안 최대 속도를 1.65배로 높인다.
- 이유: 별도의 가상 조이스틱 위치를 고정하지 않고 화면 어디서든 한 손 조작을 지원한다.
- 영향: M2에서는 부스터 자원이나 쿨다운을 추가하지 않는다. 플레이 테스트에서 무한 부스터가 사냥과 회피 선택을 약화하면 후속 밸런스 단계에서 쿨다운을 도입한다.

## D-009 — GitHub Pages Web Alpha

- 상태: 확정
- 결정: 공개 저장소 `roycekim84/fish-growth-rpg`의 `main`을 GitHub Actions로 검증하고 `https://roycekim84.github.io/fish-growth-rpg/`에 자동 배포한다.
- 이유: 모바일 빌드 이전에도 이해관계자가 별도 설치 없이 최신 플레이 상태를 확인할 수 있어야 한다.
- 영향: `main` push는 공개 Web alpha 변경으로 이어지므로 분석, 테스트, Web build 성공 후에만 Pages deploy job이 실행된다.

## D-010 — 반자동 사냥은 편의 기능으로 제한

- 상태: 확정
- 결정: M4 전투 이후 M4.5에서 즉시 포식 가능한 가까운 대상만 기본 속도의 75%로 추적하는 선택형 반자동 사냥을 실험한다.
- 이유: 모바일 반복 조작 피로를 줄이되 위험 판단, 회피, 먹고 쉬는 회복과 종변화 선택은 플레이어에게 남겨야 한다.
- 영향: 위험 대상 접근, HP 35% 이하 또는 수동 드래그 입력 시 중단하며 자동 부스터·회복·종변화와 백그라운드 진행은 지원하지 않는다.

## D-011 — NPC 종별 공격 성향

- 상태: 확정
- 결정: 공통 크기 기반 Flee 규칙을 사용하되 Chase 기준은 HunterFish 0.9, PufferFish 0.7, SmallFish 0.72의 플레이어/NPC Size 비율로 구분한다.
- 이유: 모든 큰 NPC가 같은 방식으로 추격하면 복어의 방어적 역할과 작은 물고기의 먹이 역할이 흐려진다.
- 영향: 플레이어 시작 Size 0.8에서는 SmallFish가 도망가고, PufferFish는 주로 배회하며, HunterFish는 플레이어를 추격한다.

## D-012 — M4 전투 기준선과 부활

- 상태: 확정
- 결정: 플레이어 임시 기본 스탯은 MaxHP 40, STR 3, Size 0.8로 두고 피해량은 공격자의 STR, 같은 접촉 대상의 물기 간격은 0.75초로 한다. 사망 시 월드 중앙에서 HP를 전부 회복하고 1.5초 동안 접촉 피해를 무시한다.
- 이유: M5 성장 시스템 전에도 세 가지 크기 관계와 사망 루프를 검증할 수 있는 단순하고 예측 가능한 기준선이 필요하다.
- 영향: M5에서 레벨·종 배율을 적용하더라도 판정 공식과 접촉 간격은 유지하며, 임시 기본 스탯은 데이터 기반 최종 스탯으로 교체한다.

## D-013 — 반자동 사냥 안전 경계

- 상태: 확정
- 결정: AUTO는 0.15초마다 즉시 포식 가능한 최근접 대상을 다시 고르고 수동 기본 속도의 75%로 추적한다. 150 logical pixels 안에 위험 대상이 있거나 HP가 35% 이하이면 AUTO와 자동 추진 관성을 함께 중단한다.
- 이유: 자동 추적의 남은 관성이 플레이어를 위험 대상 쪽으로 계속 밀어 넣으면 안전 중단 표시와 실제 행동이 일치하지 않는다.
- 영향: 수동 드래그와 부스터는 AUTO만 해제하고 기존 관성을 유지한다. AUTO에는 경로 계획, 위험 우회, 부스터, 회복 또는 백그라운드 진행을 추가하지 않는다.

## D-014 — M5 성장 상태와 회복 기준선

- 상태: 확정
- 결정: 플레이어는 Fullness 50/100으로 시작한다. 포식 보상은 실제 EXP·Fullness 증가량과 종별 카운트에 반영하며, 레벨 요구량은 `20 + Level * 10`을 사용한다. 회복은 1.5초 정지와 전투 후 1초를 모두 만족할 때 Fullness 5/s를 HP 8/s로 변환한다.
- 이유: 저장·종변화 전에도 먹기, 쉬기, 성장하기의 핵심 루프를 하나의 직렬화 가능한 진행 모델로 검증해야 한다.
- 영향: 레벨은 MaxHP, STR, Size, Weight를 증가시키며 현재 HP는 MaxHP 증가분만큼 오른다. 보상은 HP를 직접 회복하지 않고 Fullness 상한에서 실제 증가량만 표시한다.

## D-015 — M6 시작 종과 종변화 배율 기준선

- 상태: 확정
- 결정: 별도 NPC 데이터가 없는 시작 종 `starter_fish`는 항상 해금한다. NPC 종은 100번째 포식에서 한 번만 해금하며 SmallFish는 빠른 경량형, PufferFish는 느린 생존형, HunterFish는 강한 추격형 배율을 사용한다. 종변화는 전투 후 1초 동안 금지하고 성공 시 HP 비율을 유지한다.
- 이유: 시작 직후에도 유효한 현재 종이 필요하고, 종변화가 즉시 회복 수단이 되지 않으면서 세 종의 플레이 차이를 데이터만으로 검증해야 한다.
- 영향: 레벨·EXP·카운트·도감·해금 상태는 유지하고 외형과 MaxHP·STR·SPD·Size·Weight만 다시 계산한다. 도감과 종변화 오버레이가 열린 동안 월드는 일시 정지한다.

## D-016 — M7 로컬 저장 schema v1과 미래 버전 보호

- 상태: 확정
- 결정: `shared_preferences`의 비캐시 비동기 API에 단일 JSON 문자열을 저장한다. schema v1은 영구 진행도와 현재 HP, UTC 저장 시간을 포함하며 변경 후 700ms 디바운스와 앱 수명주기 이벤트를 함께 사용한다. 손상 데이터는 제거하지만 현재 앱보다 높은 schema는 보존하고 자동 저장을 막는다.
- 이유: Web·Android·iOS에서 같은 포맷을 사용하면서 저장 매체를 도메인에서 분리하고, 구버전 앱이 신버전 진행도를 파괴하는 상황을 막아야 한다.
- 영향: 월드 NPC, 위치, 속도, KO처럼 세션 전용 상태는 저장하지 않는다. `lastSaveTimeUtc`는 기록하지만 M7에서는 오프라인 보상을 계산하지 않는다.

## D-017 — M8 픽셀 스프라이트 원본과 런타임 분리

- 상태: 확정
- 결정: AI 생성 모델 시트와 투명화 중간본은 `art_source`에 보존하고, 게임 번들에는 검수·축소한 종별 4프레임 strip만 포함한다. Flame은 nearest-neighbor 필터로 그리며 좌향 이미지는 별도 제작하지 않고 런타임 반전한다.
- 이유: 생성 이력과 재가공 가능성을 유지하면서 Web·모바일 다운로드 크기, 메모리 사용량과 픽셀 일관성을 함께 관리해야 한다.
- 영향: 새 종은 manifest에 프레임 규격과 출처를 먼저 기록하고, 재현 가능한 빌드 도구를 거쳐 runtime 폴더에 추가한다.

## D-018 — 환경 장식은 비충돌 배경 레이어로 제한

- 상태: 확정
- 결정: M8 환경 장식은 8종 아틀라스를 결정적 위치에 렌더링하는 비충돌 배경 요소로 구현한다. 배경 타일과 props는 한 번 캐시하고 nearest-neighbor로 반복·확대하며, 로딩 실패 시 단색 바다를 유지한다.
- 이유: 장식 품질을 높이면서도 1차 프로토타입의 이동 경로, AI, 안전 스폰과 전투 밸런스를 바꾸지 않아야 한다.
- 영향: 장애물 충돌, 경로 탐색, 장식 랜덤 생성은 M8에 포함하지 않는다. 이후 지역 시스템을 추가할 때 별도 gameplay prop 컴포넌트로 분리한다.

## D-019 — 공통 코드 기반 픽셀 UI와 상태 기호

- 상태: 확정
- 결정: HUD·도감·종변화는 공통 팔레트, 이중 직각 프레임, 분절 게이지와 버튼 pressed 규칙을 공유한다. 종 초상화는 별도 중복 이미지 없이 swim strip 첫 프레임을 크롭하며, 중요 상태와 NPC 위험 관계는 색상과 기호를 함께 사용한다.
- 이유: 화면마다 중복된 장식을 제거하고 새 UI를 같은 규칙으로 확장하면서, 색상만으로 상태를 판단해야 하는 부담을 줄여야 한다.
- 영향: 본문 한글은 시스템 글꼴을 유지한다. UI 상태 아이콘은 `?`, `×`, `✓`, `■`, `◆`, 관계 마커는 `+`, `=`, `!`를 기본으로 한다.

## D-020 — 전투 감각 피드백은 선택적 계층으로 분리

- 상태: 확정
- 결정: 전투 이벤트는 파티클과 `GameFeedbackService`를 통해 전달한다. 4종의 짧은 PCM 효과음은 저장소 도구로 재생성하며 Web은 오디오, Android/iOS는 오디오와 이벤트 강도별 햅틱을 사용한다.
- 이유: 전투 감각을 강화하되 브라우저 autoplay 정책, 오디오 디코더 또는 햅틱 미지원이 게임 시작과 핵심 루프를 막아서는 안 된다.
- 영향: 오디오 프리로드는 비동기로 실행하고 모든 피드백 실패는 gameplay에서 격리한다. 테스트에서는 `SilentGameFeedbackService`를 주입하며 이후 음량·진동 설정도 같은 경계에서 추가한다.

## D-021 — 감각 피드백 채널별 중첩 제한

- 상태: 확정
- 결정: bite·playerHit은 충돌 채널 140ms, consume은 보상 채널 80ms, levelUp·unlock·defeat은 주요 채널 300ms 쿨다운을 공유한다. 시각 파티클은 제한하지 않는다.
- 이유: 여러 NPC와 동시에 접촉해도 전투 정보는 보존하면서 Web·모바일 오디오 플레이어와 햅틱 호출이 순간적으로 늘어나는 것을 막아야 한다.
- 영향: 같은 채널에서 쿨다운 안에 들어온 후속 소리·진동은 생략한다. 포식과 레벨업처럼 서로 다른 채널의 의도된 조합은 함께 재생할 수 있다.

## D-022 — 제품 방향을 탐험 어드벤처 RPG로 전환

- 상태: 확정
- 결정: 제품 기준을 `Fish Growth RPG`에서 `Fish Adventure RPG`로 전환한다. 핵심 경험은 `Explore -> Discover -> Unlock -> Explore Again`이며, 성장과 전투는 탐험을 보조하는 시스템으로 둔다.
- 이유: 기존 사냥·성장 중심 루프는 반복 포식 목표가 강해질수록 플레이 시간이 노가다로 보일 위험이 있다. 새 기준은 지역, 발견, 퀘스트, 종별 능력을 보상으로 삼아 장기 재미를 만든다.
- 영향: 기존 M0~M8 구현은 폐기하지 않는다. M9 마켓 출시 준비는 보류하고, 먼저 Region, Discovery, Quest, Species Ability 기반의 탐험 수직 절편을 만든다.

## D-023 — 종 해금은 포식 반복보다 탐험 보상 우선

- 상태: 확정
- 결정: 100마리 포식 해금은 초기 프로토타입 mechanic과 일부 collection quest로 유지하되, 장기 기준의 종 해금은 Quest, Region discovery, Boss gate, Special encounter를 우선한다.
- 이유: Species는 상위 스탯이 아니라 새 이동과 새 지역 접근을 제공해야 하므로 반복 사냥만으로 해금되면 탐험 중심 목표와 충돌한다.
- 영향: `eatenCountBySpeciesId`는 저장과 도감에 유지한다. 이후 `questUnlockId`, `abilityId`, `regionAccessTags`를 species 데이터에 추가할 수 있도록 확장한다.

## D-024 — 지역 발견은 진입과 랜드마크 기록을 분리

- 상태: 확정
- 결정: 지역 진입은 `discoveredRegionIds`, 실제 탐험은 지역별 `discoveredPointIds`로 별도 저장한다. 첫 지역 `ocean_shallows`는 네 개의 랜드마크를 모두 찾으면 발견 완료다.
- 이유: 이후 지역 이동, 퀘스트 조건, 도감 완성도, 보스 해금 조건이 하나의 단순한 포식 카운트에 묶이지 않도록 한다.
- 영향: M9에서는 진입과 근접 발견만 구현한다. 지점의 시각 마커, 지역 경계 이동, 퀘스트 보상은 후속 마일스톤에서 이 데이터를 소비한다.

## D-025 — 메인 종 해금은 탐험 퀘스트 보상으로 제공

- 상태: 확정
- 결정: 조개 안내자 누리의 첫 퀘스트는 얕은 바다 랜드마크 4곳 발견이며, 완료 보상으로 `small_fish` 종을 해금한다. 이후 수집 퀘스트는 이전 퀘스트 완료 뒤에만 수락할 수 있다.
- 이유: 같은 물고기를 100마리 먹는 행동보다 세계를 둘러보는 행동이 첫 종변화의 동기가 되어야 한다.
- 영향: 기존 100마리 해금은 collection mechanic으로 남지만 `small_fish`의 메인 해금 경로는 퀘스트가 된다. M11에서는 이 해금 종에 실제 이동 능력을 부여한다.

## D-026 — 종 능력은 패시브 지형 접근 권한부터 시작

- 상태: 확정
- 결정: 첫 능력은 별도 조작 버튼 없이 현재 종의 패시브 능력으로 동작한다. 작은 물고기는 `narrow_current`, 복어는 `coral_break` 게이트를 통과한다.
- 이유: 모바일 조작을 복잡하게 만들지 않으면서 종변화 직후 이전에는 닿지 못했던 장소에 갈 수 있다는 변화를 즉시 보여 준다.
- 영향: 얕은 바다의 `속삭이는 해류`는 작은 물고기 전용 발견 지점이 된다. 능동 스킬, 쿨다운, 전투 효과는 다음 수직 절편에서 필요할 때만 추가한다.

## D-027 — 첫 보스는 다음 지역 해금 관문으로 사용

- 상태: 확정
- 결정: `해류의 수호자`는 좁은 해류 너머 보스 아레나에 한 번만 등장한다. 처치하면 `deep_sea` 지역을 해금하고 북쪽 관문을 연다.
- 이유: 전투가 성장 반복의 목적이 아니라 다음 탐험 장소로 가는 관문이라는 Master Spec의 원칙을 첫 수직 절편부터 검증한다.
- 영향: 보스 처치와 지역 해금은 save schema v4에 보존한다. 실제 심해 맵 전환과 콘텐츠는 M13에서 추가한다.

## D-028 — 심해는 관문 통과 즉시 전환하고 현재 지역을 저장

- 상태: 확정
- 결정: 해류의 수호자 처치 뒤 열린 북쪽 관문에 닿으면 `deep_sea` 지역으로 전환한다. 현재 지역은 save schema v5에 기록해 재시작 뒤에도 같은 지역에서 이어진다.
- 이유: 지역 해금이 단순한 도감 문구로 끝나지 않고, 플레이어가 새 팔레트·새 랜드마크·새 퀘스트를 즉시 경험해야 한다.
- 영향: 첫 심해 구역은 같은 프로토타입 필드 위에서 팔레트와 발견 데이터를 교체한다. 지역별 NPC/스폰 테이블과 더 큰 지도는 다음 콘텐츠 마일스톤으로 확장한다.

## D-029 — 지역별 개체군과 퀘스트 보상 능력 경로

- 상태: 확정
- 결정: 종 데이터는 지역별 스폰 수를 선택적으로 정의한다. 심해는 작은 물고기 8, 복어 6, 사냥 물고기 12마리로 구성하며, `심해의 메아리` 퀘스트의 보상은 사냥 물고기 종변화다. 사냥 물고기의 `deep_sense` 능력은 메아리 해구를 열어 심해의 마지막 발견 지점으로 이어진다.
- 이유: 새 지역이 단지 팔레트만 다른 필드로 보이지 않게 하고, 퀘스트 보상 종이 즉시 새로운 탐험 권한을 제공해야 한다.
- 영향: 기존 `maxSpawnCount`는 지역 데이터가 없는 구역의 기본값으로 유지한다. 심해 발견률 100%는 퀘스트 완료 뒤 종변화 화면에서 사냥 물고기로 바꾼 뒤 달성할 수 있다.
