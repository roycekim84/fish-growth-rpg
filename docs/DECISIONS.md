# 결정 기록

중요한 기술·기획 결정을 짧게 누적한다. 기존 결정을 바꿀 때는 이전 기록을 삭제하지 않고 새 항목으로 대체 이유를 남긴다.

## D-001 — Flutter + Flame 사용

- 상태: 확정
- 결정: Flutter stable 위에 Flame을 사용한다.
- 이유: Android, iOS, Web을 단일 Dart 코드베이스로 지원하면서 실시간 게임 루프, 컴포넌트, 카메라, 충돌과 스프라이트 기능을 활용할 수 있다.
- 영향: UI는 Flutter overlay, 게임 월드는 Flame 중심으로 분리한다.

## D-002 — 모바일 우선, 웹은 개발·QA 보조

- 상태: 확정
- 결정: 최종 품질 기준은 Android/iOS이며 Web은 빠른 반복 테스트와 공유 가능한 데모에 사용한다.
- 이유: 웹과 모바일은 방향 고정, 오디오 캐시, 입력 및 성능 특성이 일부 다르다.
- 영향: 웹 통과만으로 모바일 마일스톤을 완료하지 않는다.

## D-003 — 세로 화면 전용

- 상태: 확정
- 결정: 플레이 화면은 portrait only로 설계한다.
- 이유: 한 손 중심의 단순한 모바일 조작과 HUD 배치를 유지한다.
- 영향: 넓은 웹 화면에서도 세로형 캔버스를 유지한다.

## D-004 — 전체 픽셀아트 스타일

- 상태: 확정
- 결정: 배경, 물고기, 이펙트, 버튼과 패널을 픽셀아트로 통일한다.
- 이유: 작은 모바일 화면에서 강한 실루엣과 일관된 제작 범위를 확보한다.
- 영향: 에셋 규격, 필터링, atlas 여백과 픽셀 정렬을 초기부터 검증한다.

## D-005 — 문서와 Git 기반 단계 진행

- 상태: 확정
- 결정: 명세, 로드맵, 결정, 작업 현황을 Markdown으로 관리하고 마일스톤별 작은 커밋으로 진행한다.
- 이유: 범위 확대와 구현 방향 흔들림을 줄이고 변경 이유를 추적한다.
- 영향: 코드 변경과 함께 관련 문서를 갱신한다.

## D-006 — 패키지 버전 고정

- 상태: 확정
- 결정: Flutter 앱의 `pubspec.lock`을 Git에 포함한다.
- 이유: 앱 프로젝트에서 개발/CI/출시 빌드의 의존성 재현성을 확보한다.
- 영향: 의존성 업그레이드는 별도 커밋과 회귀 테스트를 거친다.

## D-007 — 앱 이름과 식별자

- 상태: 확정
- 결정: 프로젝트명은 `fish_growth_rpg`, 임시 표시명은 `Fish Growth RPG`, Android application ID와 iOS bundle ID는 `com.roycekim.fishgrowthrpg`를 사용한다.
- 이유: M1 프로젝트 생성과 양 플랫폼 설정에 사용할 안정적인 기본값이 필요하다.
- 영향: 실제 출시 이름을 바꾸더라도 패키지 식별자는 기존 설치 및 스토어 등록과 연결되므로 신중하게 변경한다.

## D-008 — 드래그 이동과 1차 부스터 정책

- 상태: 확정
- 결정: 최초 터치 지점을 앵커로 삼아 드래그 방향으로 가속하며, 손을 떼면 추진을 중단하고 관성 감속한다. 부스터는 버튼을 누르는 동안 최대 속도를 1.65배로 높인다.
- 이유: 별도의 가상 조이스틱 위치를 고정하지 않고 화면 어디서든 한 손 조작을 지원한다.
- 영향: M2에서는 부스터 자원이나 쿨다운을 추가하지 않는다. 플레이 테스트에서 무한 부스터가 사냥과 회피 선택을 약화하면 후속 밸런스 단계에서 쿨다운을 도입한다.

## D-009 — GitHub Pages Web Alpha

- 상태: 확정
- 결정: 공개 저장소 `roycekim84/fish-growth-rpg`의 `main`을 GitHub Actions로 검증하고 `https://roycekim84.github.io/fish-growth-rpg/`에 자동 배포한다.
- 이유: 모바일 빌드 이전에도 이해관계자가 별도 설치 없이 최신 플레이 상태를 확인할 수 있어야 한다.
- 영향: `main` push는 공개 Web alpha 변경으로 이어지므로 분석, 테스트, Web build 성공 후에만 Pages deploy job이 실행된다.

## D-010 — 반자동 사냥은 편의 기능으로 제한

- 상태: 확정
- 결정: M4 전투 이후 M4.5에서 즉시 포식 가능한 가까운 대상만 기본 속도의 75%로 추적하는 선택형 반자동 사냥을 실험한다.
- 이유: 모바일 반복 조작 피로를 줄이되 위험 판단, 회피, 먹고 쉬는 회복과 종변화 선택은 플레이어에게 남겨야 한다.
- 영향: 위험 대상 접근, HP 35% 이하 또는 수동 드래그 입력 시 중단하며 자동 부스터·회복·종변화와 백그라운드 진행은 지원하지 않는다.

## D-011 — NPC 종별 공격 성향

- 상태: 확정
- 결정: 공통 크기 기반 Flee 규칙을 사용하되 Chase 기준은 HunterFish 0.9, PufferFish 0.7, SmallFish 0.72의 플레이어/NPC Size 비율로 구분한다.
- 이유: 모든 큰 NPC가 같은 방식으로 추격하면 복어의 방어적 역할과 작은 물고기의 먹이 역할이 흐려진다.
- 영향: 플레이어 시작 Size 0.8에서는 SmallFish가 도망가고, PufferFish는 주로 배회하며, HunterFish는 플레이어를 추격한다.

## D-012 — M4 전투 기준선과 부활

- 상태: 확정
- 결정: 플레이어 임시 기본 스탯은 MaxHP 40, STR 3, Size 0.8로 두고 피해량은 공격자의 STR, 같은 접촉 대상의 물기 간격은 0.75초로 한다. 사망 시 월드 중앙에서 HP를 전부 회복하고 1.5초 동안 접촉 피해를 무시한다.
- 이유: M5 성장 시스템 전에도 세 가지 크기 관계와 사망 루프를 검증할 수 있는 단순하고 예측 가능한 기준선이 필요하다.
- 영향: M5에서 레벨·종 배율을 적용하더라도 판정 공식과 접촉 간격은 유지하며, 임시 기본 스탯은 데이터 기반 최종 스탯으로 교체한다.

## D-013 — 반자동 사냥 안전 경계

- 상태: 확정
- 결정: AUTO는 0.15초마다 즉시 포식 가능한 최근접 대상을 다시 고르고 수동 기본 속도의 75%로 추적한다. 150 logical pixels 안에 위험 대상이 있거나 HP가 35% 이하이면 AUTO와 자동 추진 관성을 함께 중단한다.
- 이유: 자동 추적의 남은 관성이 플레이어를 위험 대상 쪽으로 계속 밀어 넣으면 안전 중단 표시와 실제 행동이 일치하지 않는다.
- 영향: 수동 드래그와 부스터는 AUTO만 해제하고 기존 관성을 유지한다. AUTO에는 경로 계획, 위험 우회, 부스터, 회복 또는 백그라운드 진행을 추가하지 않는다.

## D-014 — M5 성장 상태와 회복 기준선

- 상태: 확정
- 결정: 플레이어는 Fullness 50/100으로 시작한다. 포식 보상은 실제 EXP·Fullness 증가량과 종별 카운트에 반영하며, 레벨 요구량은 `20 + Level * 10`을 사용한다. 회복은 1.5초 정지와 전투 후 1초를 모두 만족할 때 Fullness 5/s를 HP 8/s로 변환한다.
- 이유: 저장·종변화 전에도 먹기, 쉬기, 성장하기의 핵심 루프를 하나의 직렬화 가능한 진행 모델로 검증해야 한다.
- 영향: 레벨은 MaxHP, STR, Size, Weight를 증가시키며 현재 HP는 MaxHP 증가분만큼 오른다. 보상은 HP를 직접 회복하지 않고 Fullness 상한에서 실제 증가량만 표시한다.

## D-015 — M6 시작 종과 종변화 배율 기준선

- 상태: 확정
- 결정: 별도 NPC 데이터가 없는 시작 종 `starter_fish`는 항상 해금한다. NPC 종은 100번째 포식에서 한 번만 해금하며 SmallFish는 빠른 경량형, PufferFish는 느린 생존형, HunterFish는 강한 추격형 배율을 사용한다. 종변화는 전투 후 1초 동안 금지하고 성공 시 HP 비율을 유지한다.
- 이유: 시작 직후에도 유효한 현재 종이 필요하고, 종변화가 즉시 회복 수단이 되지 않으면서 세 종의 플레이 차이를 데이터만으로 검증해야 한다.
- 영향: 레벨·EXP·카운트·도감·해금 상태는 유지하고 외형과 MaxHP·STR·SPD·Size·Weight만 다시 계산한다. 도감과 종변화 오버레이가 열린 동안 월드는 일시 정지한다.
